# 工程方法论

## 架构决策记录（ADR）

每个重要技术决策必须留记录。"重要"的定义：影响 >1 个模块、不可轻易回退、或团队有分歧。

### ADR 模板
```markdown
# ADR-{序号}: {决策标题}

- 日期: {YYYY-MM-DD}
- 状态: 提议 / 采纳 / 废弃 / 替代(被ADR-XX替代)
- 决策者: product-lead + fullstack-dev

## 背景
{为什么需要做这个决策？当前的问题是什么？}

## 决策
{我们选择了什么方案？}

## 备选方案
| 方案 | 优点 | 缺点 | 排除原因 |
|------|------|------|---------|
| A | | | |
| B(采纳) | | | — |
| C | | | |

## 影响
- 影响模块: {列出}
- 迁移成本: 高/中/低/无
- 风险: {可能的问题}
- 技术债: {引入了什么债/减少了什么债}

## 后续
- [ ] {需要做的事}
```

### ADR 存放位置
`shared/products/{产品}/adrs/` 目录，或写在 architecture.md 的 ## 决策记录 章节。

### 触发 ADR 的场景
- 新增/替换框架或主要依赖
- 数据库 schema 重大变更
- API 破坏性变更
- 新增微服务/模块拆分
- 认证/授权方案变更
- 部署架构变更

## 代码 Review 清单

### 通用检查（所有PR）
- [ ] **功能正确**: 满足验收标准，边界情况处理
- [ ] **无回退**: 不破坏已有功能（跑现有测试）
- [ ] **命名清晰**: 变量/函数/类名可读、一致
- [ ] **无硬编码**: 配置项走 env/config，不写死
- [ ] **错误处理**: try-catch 合理，用户友好错误提示
- [ ] **日志合理**: 关键操作有日志，无敏感信息泄露

### 安全检查
- [ ] **输入验证**: 所有用户输入已校验/转义
- [ ] **SQL注入**: 使用参数化查询/ORM，无拼接SQL
- [ ] **XSS**: 输出已转义，dangerouslySetInnerHTML 有理由
- [ ] **CSRF**: 表单有token
- [ ] **认证/授权**: 路由权限检查，无越权访问
- [ ] **敏感数据**: 密码哈希、token不明文、日志无敏感信息
- [ ] **依赖安全**: 无已知漏洞依赖（npm audit / composer audit）

### 性能检查
- [ ] **N+1查询**: 列表页是否 eager load
- [ ] **索引**: 新增查询条件是否有对应索引
- [ ] **大数据**: 分页是否正确，无全表扫描
- [ ] **缓存**: 高频读取是否命中缓存
- [ ] **前端**: 组件是否合理 memo/懒加载，bundle size

### 可维护性检查
- [ ] **单一职责**: 函数/组件不超过 100 行（超了要拆）
- [ ] **重复代码**: DRY，有无可抽取的公共逻辑
- [ ] **测试**: 新功能有测试，改动不破坏已有测试
- [ ] **文档**: 复杂逻辑有注释，API 变更更新文档
- [ ] **兼容性**: 数据库 migration 可回滚

### Review 严格度（按任务复杂度）
| 复杂度 | Review 方式 |
|--------|------------|
| Simple | 自检通过即可（功能ok + lint pass + tests pass） |
| Medium | 过通用 + 安全清单 |
| Complex | 全部清单 + 架构影响评估 + 单独QA |

## 技术债评估框架

### 技术债分类
| 类型 | 描述 | 示例 |
|------|------|------|
| **架构债** | 系统结构不合理 | 单体过大、循环依赖、缺中间层 |
| **代码债** | 代码质量差 | 重复代码、过长函数、magic number |
| **依赖债** | 过时/不安全依赖 | 大版本落后、弃维库、安全漏洞 |
| **测试债** | 测试覆盖不足 | 无测试、只有happy path、无E2E |
| **文档债** | 文档缺失/过时 | API无文档、README过时、部署步骤缺失 |
| **基础设施债** | DevOps/部署问题 | 手动部署、无CI/CD、无监控 |

### 技术债评分（DORA-lite）
| 因子 | 说明 | 分值 |
|------|------|:----:|
| **影响范围** | 影响多少模块/用户 | 1-5 |
| **修复难度** | 工时估算 | 1-5（5=最难） |
| **恶化速度** | 不修会不会越来越糟 | 1-5（5=最快恶化） |
| **阻塞程度** | 是否阻碍新功能开发 | 1-5（5=严重阻塞） |

**技术债优先级 = 影响范围 × 恶化速度 × 阻塞程度 / 修复难度**

分数越高越优先处理。

### 技术债登记格式
```markdown
## TD-{序号}: {标题}
- 类型: 架构/代码/依赖/测试/文档/基础设施
- 位置: {文件/模块}
- 评分: 影响={} 难度={} 恶化={} 阻塞={} → 优先级={}
- 描述: {具体问题}
- 修复方案: {怎么修}
- 预估工时: {}
- 状态: 登记/计划中/修复中/已完成
```

### 技术债治理规则
- **每个sprint分配 20% 时间还债**（product-lead 认可）
- 新功能不得引入同类型的新债（"先还再借"）
- 每月做一次技术债 review，更新评分和优先级
- 高恶化速度(≥4)的债立即排期，不等sprint

## Git 工作流

### 分支策略（简化 Git Flow）
```
main          ← 生产分支，只接受 merge
  └── develop ← 开发主线
       ├── feature/xxx  ← 功能分支
       ├── fix/xxx      ← 修复分支
       └── hotfix/xxx   ← 紧急修复（从main拉，合回main+develop）
```

### Commit 规范
```
<type>(<scope>): <subject>

type: feat|fix|refactor|docs|style|test|chore|perf
scope: 模块名（可选）
subject: 祈使句，不超过72字符
```
