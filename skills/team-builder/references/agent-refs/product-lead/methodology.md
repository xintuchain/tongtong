# 产品方法论

## 需求成熟度模型

| 级别 | 状态 | 能否派发 | 描述 |
|------|------|:--------:|------|
| **L0 原始想法** | 💭 raw | ❌ | CEO拍脑袋/用户随口说/竞品有 |
| **L1 用户验证** | 🔍 validated | ❌ | 确认真实用户痛点（有数据支撑） |
| **L2 方案定义** | 📋 defined | ❌ | 用户故事+验收标准+影响范围+技术可行性 |
| **L3 就绪** | ✅ ready | ✅ | L2+优先级确认+技术评审+无未解决依赖 |

## 需求验证检查清单（L0→L1）

1. **谁要的？** 真实用户还是假想用户？多少用户有此需求？
2. **频率多高？** 偶尔还是每天？（频率×痛感=优先级）
3. **现在怎么办？** 用户当前替代方案？够不够用？
4. **数据说什么？** data-analyst 有反馈数据？行为数据？
5. **不做会怎样？** 6个月不做，丢多少用户/收入？

## 需求拒绝场景
- CEO说"加个XX" → "哪些用户反馈过？我查数据"
- 竞品上了新功能 → "我们用户需要吗？还是竞品试错？"
- 内部觉得"应该有" → "有数据支持吗？先放backlog"
- 需求模糊 → "具体哪个场景？用户什么问题？"

## RICE 优先级打分

| 因子 | 说明 | 范围 |
|------|------|------|
| **R**each | 影响用户数（月活%） | 0.5-10 |
| **I**mpact | 单个用户影响程度 | 0.25(低)-3(巨大) |
| **C**onfidence | 估计信心 | 50%-100% |
| **E**ffort | 人·周 | 0.5-10 |

**RICE = (R×I×C) / E**

## 需求模板（L2 产出）

```markdown
## 需求: {标题}
- ID: REQ-{YYYYMMDD}-{序号}
- 来源: {用户反馈/数据洞察/CEO指示/竞品分析}
- 用户故事: 作为{角色}，我想{操作}，以便{价值}
- 验收标准: [列表]
- 数据支撑: {feedback数量/行为数据}
- 影响范围: {涉及模块}
- RICE: R=_ I=_ C=_ E=_ → 分数: _
- 状态: L0→L1→L2→L3
```

## 紧急例外（可跳过验证）
- 🔴 P0 线上故障 / 安全漏洞
- 🟡 CEO明确标注"紧急"且给出理由
- 跳过时标注 `[紧急跳过验证]`，事后补
