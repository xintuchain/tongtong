# 团队对抗性内容审查框架

> **建立批判性思维文化，让每个产出都经过严格审视**

---

## 背景

数字团队目前产出的内容缺乏对抗性审查机制。为了提升内容质量和团队批判性思维，我们需要建立一套系统化的对抗性内容审查（Adversarial Content Review）流程。

### 核心原则

1. **不信任原则** — 对每个结论、每个数据、每个假设持怀疑态度
2. **角色对立原则** — 安排专门的角色扮演"攻击者"，挑战产出
3. **迭代改进原则** — 审查不是一次性事件，而是循环改进过程

## 团队角色与职责

### 红队（攻击方）

| 角色 | 负责成员 | 核心职责 |
|------|----------|---------|
| 🔴 **质疑者** (Devil's Advocate) | 慧研 | 挑战假设、深挖漏洞、寻找反例 |
| 🔧 **技术专家** | 慧选 | 技术准确性、可行性、安全性审查 |
| 📢 **受众测试员** | 慧联 | 受众理解、传播效果、可读性 |
| 🛡️ **安全审计员** | 慧维 | 风险评估、偏见检测、合规性 |

### 蓝队（防守方）

| 角色 | 负责成员 | 核心职责 |
|------|----------|---------|
| 🔵 **产出者** | 各任务负责人 | 理解批评、回应质疑、修改完善 |
| 🔵 **协调者** | 桐桐 | 主持审查、平衡观点、推动决策 |

## 审查流程

```
┌─────────────────────────────────────────────────────────────┐
│                      产出提交阶段                           │
│  产出者完成初稿 → 提交至 Discord 审查频道                    │
└─────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────┐
│                      红队审查阶段                           │
│  并行执行四个审查维度：                                     │
│  ├── 质疑者：假设与逻辑审查                                 │
│  ├── 技术专家：技术与可行性审查                             │
│  ├── 受众测试员：受众理解度审查                             │
│  └── 安全审计员：风险与偏见审查                             │
└─────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────┐
│                      产出者回应阶段                         │
│  产出者回应每条质疑，分类处理：                             │
│  ├── 接受：修改内容                                        │
│  ├── 拒绝：提供充分理由                                     │
│  └── 待定：标记为开放问题                                    │
└─────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────┐
│                      协调者裁决阶段                         │
│  桐桐主持讨论，平衡红蓝双方观点                             │
│  最终决策：接受 / 拒绝 / 需重大修改                         │
└─────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────┐
│                      最终交付阶段                           │
│  修改后再次提交 → 如有必要再次审查 → 最终发布                │
└─────────────────────────────────────────────────────────────┘
```

## 审查问题清单

### 质疑者必问清单

```
□ 这个结论的核心假设是什么？
□ 这些假设能被推翻吗？最薄弱的地方在哪？
□ 有哪些反面的证据或案例被忽略了？
□ 如果这个结论是错的，最可能的原因是什么？
□ 我们是在相关还是因果？混淆了吗？
□ 是否存在确认偏误（只找支持自己观点的证据）？
□ 样本量足够吗？能代表整体吗？
□ 因果链完整吗？有没有遗漏中间环节？
□ 是否有"事后归因"谬误？
□ 结论是否超出了证据所能支持的范围？
```

### 技术专家必问清单

```
□ 技术细节描述准确吗？有参考文献支持吗？
□ 实现方案技术上可行吗？有什么障碍？
□ 数据来源可靠吗？采集方法科学吗？
□ 是否有技术债务或隐患？
□ 性能、安全性、可扩展性如何？
□ 是否有更好的替代方案被忽略了？
□ 技术术语使用正确吗？
□ 代码/流程是否存在漏洞？
```

### 受众测试员必问清单

```
□ 目标读者能理解这篇文章吗？
□ 专业术语是否需要解释？
□ 核心信息够清晰吗？一句话能说清楚吗？
□ 有没有歧义或容易误解的地方？
□ 结构是否清晰？逻辑链条完整吗？
□ 开头够吸引人吗？读者会继续读下去吗？
□ 适合在哪些渠道传播？
□ 有没有文化敏感性问题？
□ 情感基调合适吗？太激进还是太温和？
```

### 安全审计员必问清单

```
□ 有没有潜在的法律风险？
□ 是否有侵犯隐私的风险？
□ 内容是否有歧视性或偏见？
□ 是否有误导性信息？
□ 是否泄露了敏感信息？
□ 结论是否过于绝对？有没有加"可能"或"或许"？
□ 是否过度承诺？
□ 有没有利益冲突被披露？
□ 是否可以被恶意利用？
□ 可复制性和可扩展性如何？
```

## 审查等级

### 等级一：快速审查（5分钟）
适用于：日常消息、简单回复、快速更新
- 只问最关键的3个问题
- 由产出者自行完成

### 等级二：标准审查（15分钟）
适用于：研究报告、方案设计、正式文档
- 完整四个维度审查
- 红队成员轮流担任

### 等级三：深度审查（30分钟以上）
适用于：战略决策、重要报告、对外发布内容
- 完整红队审查 + 蓝队回应 + 协调者裁决
- 可能需要多轮迭代

## 实施指南

### Discord 审查频道

建议在 Discord 建立专门的审查频道：
- `#对抗性审查` — 提交审查请求
- `#审查记录` — 存储审查历史

### 审查请求格式

```
📋 审查请求

**产出类型**：[研究报告/方案/回复/其他]
**目标受众**：[谁会看这个内容]
**紧急程度**：[快速/标准/深度]
**链接/附件**：[内容位置]

**自评（产出者填写）**：
- 我认为最薄弱的地方是：___
- 我最不确定的是：___
```

### 审查回应格式

```
🔍 审查意见

**审查员**：@慧研/慧选/慧联/慧维
**审查类型**：质疑/技术/受众/安全
**严重程度**：🔴严重 / 🟡中等 / 🟢建议

**具体意见**：
[你的具体批评或建议]

**建议处理**：接受 / 拒绝 / 待讨论
**理由**：[为什么这样处理]
```
